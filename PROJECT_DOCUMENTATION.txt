1. PROJECT OVERVIEW
================================================================================

NextBoard.ai is a full-stack web application that transforms meeting recordings 
and transcripts into structured, AI-generated action boards. It eliminates the 
need for PowerPoint summaries and manual follow-ups by automatically extracting 
actionable items from meetings.

Key Value Proposition:
- Upload meeting transcripts (TXT, PDF) or recordings
- AI-powered extraction of actions, decisions, follow-ups, and updates
- Beautiful Kanban board interface with drag-and-drop
- Multiple view modes (Kanban, Calendar, Focus, Priority Matrix)
- Time tracking and analytics
- Multi-user support with authentication
- Email notifications

Production URLs:
- Primary: https://next-board.io
- Backup: https://next-board.ai

4.1 PREREQUISITES
-----------------
- Node.js 18+ (recommended: Node.js 20)
- npm or yarn
- PostgreSQL database (or Railway account)
- Anthropic API Key (optional - demo mode available)
- Clerk account (for authentication)
- Resend account (for email notifications)

4.2 LOCAL DEVELOPMENT SETUP
---------------------------

Step 1: Clone Repository
  git clone <your-repo-url>
  cd next-board.io

Step 2: Install Dependencies
  npm install

Step 3: Set Up Environment Variables
  Create .env file in root directory (see section 5 for details)

Step 4: Set Up Database
  # For local SQLite (development)
  npx prisma db push

  # OR for PostgreSQL (production-like)
  # Set DATABASE_URL in .env first
  npx prisma migrate dev

Step 5: Generate Prisma Client
  npx prisma generate

Step 6: Start Development Server
  npm run dev

  Server will start on http://localhost:3005

6.2 RELATIONSHIPS
-----------------
- Meeting has many MeetingCard (one-to-many)
- MeetingCard has many CardActivity (one-to-many)
- MeetingCard has many CardAttachment (one-to-many)
- MeetingCard has many StatusHistoryEntry (one-to-many)
- Meeting can belong to MeetingSeries (many-to-one)
- Meeting can link to previous/next Meeting (self-referential)
- User has one NotificationPreferences (one-to-one)
- User has many Notification (one-to-many)

7.2 ENDPOINTS
-------------

POST /api/upload
  Description: Upload a meeting transcript or recording file
  Content-Type: multipart/form-data
  Body:
    - file (File, required)
    - title (string, optional)
    - participants (string, optional)
    - date (string, optional)
  Response: { success: boolean, file: {...}, metadata: {...} }
  Supported Types: .txt, .pdf, .vtt, .docx, .mp3, .mp4, .wav

POST /api/process
  Description: Process uploaded transcript with AI to extract action items
  Content-Type: application/json
  Body:
    - filepath (string, required)
    - title (string, optional)
    - participants (string, optional)
    - date (string, optional)
    - transcript (string, optional) - Raw transcript text
  Response: { success: boolean, meeting: {...} }

GET /api/board/:id
  Description: Retrieve a specific meeting board with all cards
  Path Params: id (string) - Meeting ID
  Response: { id, title, summary, createdAt, cards: [...] }

GET /api/boards
  Description: List all meeting boards for the authenticated user
  Response: { meetings: [...] }

PUT /api/card/:id
  Description: Update a card's details
  Path Params: id (string) - Card ID
  Content-Type: application/json
  Body:
    - status (string, optional)
    - owner (string, optional)
    - dueDate (string, optional)
    - summary (string, optional)
    - context (string, optional)
    - priority (string, optional)
    - timeEstimate (number, optional)
  Response: { success: boolean, card: {...} }

DELETE /api/card/:id
  Description: Delete a card permanently
  Path Params: id (string) - Card ID
  Response: { success: boolean, message: string }

POST /api/cards/:id/notes
  Description: Add a note to a card
  Path Params: id (string) - Card ID
  Content-Type: application/json
  Body: { content: string }
  Response: { success: boolean, activity: {...} }

POST /api/cards/:id/summary
  Description: Generate AI summary for a completed card
  Path Params: id (string) - Card ID
  Response: { success: boolean, summary: string }

GET /api/analytics
  Description: Get analytics data for dashboard
  Response: { staleItems, autoEscalated, avgTime, totalTime, overdue, dueSoon, thisWeek, doneToday }

POST /api/cron/auto-priority-escalation
  Description: Cron job endpoint for auto-priority escalation
  Headers: { "x-cron-secret": string }
  Response: { success: boolean, updated: number }

7.3 ERROR RESPONSES
-------------------
All endpoints return errors in this format:
  {
    "error": "Error message",
    "details": "Optional detailed error information",
    "hint": "Optional helpful hint"
  }

Common Status Codes:
  400 - Bad Request (Invalid input)
  401 - Unauthorized (Authentication required)
  404 - Not Found (Resource doesn't exist)
  500 - Internal Server Error (Server error)

7.4 AUTHENTICATION
------------------
All API endpoints (except /api/upload and /api/process) require Clerk 
authentication. Include the Clerk session token in requests.

11. TROUBLESHOOTING
================================================================================

11.1 COMMON ISSUES
------------------

Issue: "Failed to upload file"
  Solution:
    - Check file size (max 50MB)
    - Verify file format is supported (.txt, .pdf, .vtt, .docx)
    - Check file permissions
    - Verify uploads directory exists and is writable

Issue: "Failed to process transcript"
  Solution:
    - Ensure ANTHROPIC_API_KEY is set correctly
    - Check Anthropic API quota/billing
    - Verify transcript has actual content (not empty)
    - Check API logs for detailed error messages
    - Try demo mode (remove API key)

Issue: "Database connection errors"
  Solution:
    - Verify DATABASE_URL is correct
    - Check PostgreSQL is running (local) or Railway database is active
    - Run: npx prisma migrate deploy
    - Check database credentials
    - Verify network connectivity

Issue: "Drag-and-drop not working"
  Solution:
    - Clear browser cache
    - Check console for JavaScript errors
    - Ensure React version is compatible
    - Verify @dnd-kit packages are installed
    - Check z-index conflicts

Issue: "Authentication not working"
  Solution:
    - Verify Clerk API keys are set correctly
    - Check NEXT_PUBLIC_CLERK_* variables are set
    - Verify Clerk application is active
    - Check browser console for Clerk errors
    - Ensure sign-in/sign-up URLs are correct

Issue: "Theme toggle not working"
  Solution:
    - Clear localStorage
    - Check ThemeContext is properly set up
    - Verify tailwind.config.ts has darkMode: 'class'
    - Check for conflicting theme systems

Issue: "Email notifications not sending"
  Solution:
    - Verify RESEND_API_KEY is set
    - Check Resend account has credits
    - Verify RESEND_FROM_EMAIL is valid
    - Check email logs in Resend dashboard

12. ARCHITECTURE
================================================================================

12.1 APPLICATION ARCHITECTURE
------------------------------

Frontend (Next.js App Router):
  - Server Components for initial render
  - Client Components for interactivity
  - API Routes for backend logic
  - Middleware for authentication

State Management:
  - Zustand for global state (meetings, boards)
  - React Context for theme management
  - Local state for component-specific data

Data Flow:
  1. User uploads transcript → /api/upload
  2. File saved → /api/process called
  3. AI processes transcript → Cards extracted
  4. Cards saved to database
  5. Board page fetches cards → Displayed in Kanban

12.2 DATABASE ARCHITECTURE
--------------------------
- PostgreSQL (Production) / SQLite (Local)
- Prisma ORM for type-safe queries
- Migrations for schema changes
- Indexes for performance optimization

12.3 AI PROCESSING FLOW
-----------------------
1. Transcript uploaded
2. Text extracted (PDF parsing if needed)
3. Sent to Anthropic Claude API
4. AI extracts structured JSON:
   - Meeting title
   - Summary
   - Cards (type, summary, owner, due date, etc.)
5. JSON parsed and validated
6. Cards saved to database
7. Board created/updated

12.4 AUTHENTICATION FLOW
------------------------
1. User visits protected route
2. Clerk middleware checks authentication
3. If not authenticated → Redirect to /sign-in
4. After sign-in → Redirect to original page
5. User data fetched using Clerk user ID
6. All queries filtered by userId
